Project-specific Claude Code configuration including custom AI agents, status line display, MCP server integrations, and permission settings. Tailors Claude Code's behavior to Household Hub's offline-first, event-sourced architecture.

## MCP Server Integration

### What is MCP?

**MCP (Model Context Protocol):** Standard protocol for connecting AI assistants to external tools and data sources

**In Household Hub:**

- Provides access to library documentation (Context7)
- Enables UI component scaffolding (shadcn)
- Extends Claude's capabilities beyond code

### Enabled Servers

#### 1. shadcn MCP Server

**Purpose:** Interact with shadcn/ui component library

**Available operations (.claude/settings.local.json:22-25):**

- `mcp__shadcn__get_project_registries` - List configured registries
- `mcp__shadcn__list_items_in_registries` - Browse components
- `mcp__shadcn__search_items_in_registries` - Search by keyword
- `mcp__shadcn__get_item_examples_from_registries` - View usage examples
- `mcp__shadcn__view_items_in_registries` - Detailed component info

**Use cases:**

- Adding new shadcn/ui components
- Viewing component examples
- Checking component APIs

**Example:**

> "Search for a calendar component in shadcn registry"

#### 2. Context7 Server

**Purpose:** Fetch up-to-date library documentation

**Available operations (.claude/settings.local.json:4-5):**

- `mcp__context7__resolve-library-id` - Find library by name
- `mcp__context7__get-library-docs` - Fetch documentation

**Use cases:**

- Looking up latest TanStack Router API
- Checking Supabase client methods
- Verifying Dexie.js syntax

**Example:**

> "Get the latest documentation for TanStack Query v5"

## Permission System

### Auto-Approved Tools (.claude/settings.local.json:3-27)

**Purpose:** Tools Claude can use without asking permission

**Categories:**

1. **MCP Operations:**
   - Context7 documentation lookup
   - shadcn component search

2. **Bash Commands:**
   - `npm run lint`, `npm run build`, `npm test`
   - `npx eslint`, `npx tsc`
   - `wc`, `echo`, `awk`, `cat` (read-only utilities)

3. **File Access:**
   - `Read(//Users/jasonacido/Downloads/**)` - Import data files

**Why permission system?**

- Prevents accidental destructive commands
- Speeds up common operations
- User retains control over sensitive actions

**What requires permission:**

- File writes/edits
- Git operations (commit, push)
- Installing dependencies
- Modifying configuration

## Troubleshooting

### Issue: Agent not being invoked

**Check:**

1. Is task description clear about domain? (e.g., "database schema")
2. Is agent file present in `.claude/agents/`?
3. Try explicit request: "Use the supabase-schema-architect agent to..."

**Resolution:** Rephrase request to align with agent specialization

### Issue: Status line not displaying

**Check:**

1. Is `statusline.sh` executable? (`ls -l .claude/statusline.sh`)
2. Is bash available? (`which bash`)
3. Test script manually: `./.claude/statusline.sh`

**Fix:**

```bash
chmod +x .claude/statusline.sh
```

### Issue: MCP server tools not working

**Check:**

1. Is server enabled in `settings.local.json`?
2. Is MCP server installed globally?
3. Check Claude Code logs for errors

**Resolution:**

- Verify `enabledMcpjsonServers` includes server name
- Check global `~/.config/claude/mcp.json` configuration

### Issue: Permission denied for auto-approved tool

**Cause:** Tool not in `permissions.allow` list

**Fix:** Add tool pattern to `settings.local.json`:

```json
{
  "permissions": {
    "allow": ["Bash(your-command:*)"]
  }
}
```

## Best Practices

### 1. Keep Agents Focused

**Good:** One agent per domain (database, currency, UI)
**Bad:** Generic "helper" agent with mixed responsibilities

**Benefits:**

- Clear invocation criteria
- Predictable behavior
- Easier to maintain

### 2. Use Explanatory Mode for Learning

**When to use:**

- Learning new architecture patterns
- Onboarding new developers
- Complex implementation tasks

**When to switch to Concise:**

- Repetitive tasks (many similar files)
- Time-sensitive work
- Familiar patterns

### 3. Leverage MCP for Latest Docs

**Use MCP when:**

- Library recently updated
- Unsure about API syntax
- Need code examples

**Use local docs when:**

- Project-specific patterns (DATABASE.md)
- Architecture decisions (DECISIONS.md)
- Custom implementations

### 4. Customize Permissions by Role

**Developer permissions:**

- Auto-approve: lint, test, read
- Require confirmation: write, commit, push

**CI/CD permissions:**

- Auto-approve: read-only operations
- Deny: all write operations

## Security Considerations

### Status Line Script

**Potential risk:** Executes bash script on every interaction

**Mitigation:**

- Script is simple (no network calls, no file writes)
- Committed to git (reviewed by team)
- User can disable: Remove from `settings.json`

**Recommendation:** Review script before use, especially if forked from external repo

### Permission System

**Important:** Permissions are user-controlled, not enforced by Claude

**Best practices:**

- Only auto-approve read-only operations
- Require confirmation for writes
- Never auto-approve `rm`, `sudo`, or destructive commands

**Current config is safe:**

- Lint, test, build only
- No file system modifications
- No git operations

### Agent Context

**Agents have access to:**

- All project documentation
- File system (via Read tool)
- Bash commands (via Bash tool)

**What agents cannot do:**

- Access network directly
- Modify files without user permission
- Execute arbitrary code

**Recommendation:** Review agent markdown files before enabling
